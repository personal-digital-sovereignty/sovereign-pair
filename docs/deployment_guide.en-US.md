# Sovereign Pair - Cloud Deployment Guide

**Infrastructure Version:** 3.3.0
**Environmental Target:** Oracle Cloud Infrastructure (OCI) / On-Premises Docker
**Prerequisites:** Terraform (OpenTofu) >= 1.6.0, Docker Engine >= 24.0.0

This document provides the operational deployment guidelines for the Sovereign Pair platform. It is intended for DevOps Engineers responsible for instantiating the Cibrid RAG mesh (Cloud Orchestrator Node + Physical Inferential Node) using Infrastructure as Code (IaC) approaches.

---

## 1. Container Architecture (Data Plane)

Isolated provisioning on the node occurs through the declarative Docker Compose specification. The standard `enterprise` topology segregates services on an internal bridge network (`sovereign-net`).

### 1.1 Core Services
1. **`sovereign-db` (PostgreSQL 15+):** Manages system metadata, Multi-Tenant control, and traceability of Vault Sessions.
2. **`sovereign-chroma` (ChromaDB):** The Vector Data Store. Hosts semantic matrices on block disks attached via *Bind Mounts* or persistent Named Volumes.
3. **`sovereign-api` (FastAPI / Uvicorn):** Logical core. Orchestrates RAG injection and routing of LLMs (Ollama) under the HTTP REST protocol and Server-Sent Events (SSE).
4. **`sovereign-web` (Vue3 / Nginx Proxy):** Delivers pre-compiled statically optimized SPA artifacts (*dist*).

### 1.2 Network Bodyguards (Ingress/Egress)
1. **`sovereign-caddy` (M2M Reverse Proxy):** Intercepts 443 traffic. Resolves domain names on the intranet by issuing natively signed TLS certificates for zero-config security.
2. **`sovereign-tailscale` (VPN Sidecar):** The SDN (Software-Defined Network) node. Connects the container DMZ to the Tailnet's WireGuard-based encrypted Mesh, negating the physical exposure of public ports on the Cloud provider.

## 2. Cloud Operation Standards (OCI A1 Flex)

Hosting the Orchestration Plane is primarily based on the Core Computes service of the *Oracle Cloud Infrastructure (OCI)*, leveraging Ampere (ARM64) OCPUs.

### 2.1 Node Specifications (Target Baseline)
- **Shape:** `VM.Standard.A1.Flex`
- **CPU:** 4 to 6 OCPUs (Depending on PAYG or Always Free Tier)
- **RAM:** 24 GB
- **Operating System:** Ubuntu 22.04 LTS (Canonical) or Oracle Linux 9.

### 2.2 Automation Trigger (Cloud-Init)
State injection on the Oracle VM is entirely automated by Terraform. The `infra/terraform/compute.tf` file passes a Base64 payload (*cloud-init.yaml*) responsible for:
1. Updating Certificate Chains and base `apt-get`.
2. Transparent installation of native Docker Engine.
3. Unsupervised Tailscale authentication, instantly associating the machine to its private subnet without demanding pre-existing key pairs other than the ephemeral token (`TS_AUTHKEY`).

### 2.3 Host Capacity Handling (OCI 500 Error)
Due to extreme constraint on the grouping of Ampere Racks in High-Demand Datacenters (e.g., us-ashburn-1), automated deployments suffer from the recurring `Out of host capacity` message.

**Operational Solution:**
Engineers must delegate the creation task to the autonomous script. The `retry_deploy.sh` binary wraps the `tofu apply -auto-approve` instruction in an optimized loop, snatching Hardware slots that free up on the provider's edge every 2 minutes. **SRE Warning:** Cancel manual deletions (Destroy). Modify pure configurations via block replacement (In-Place Updates) in the *tfvars* so you don't drop to the back of the Tofu request queue.

## 3. Zero-Trust Network Topology and TLS

Exposure to external surfaces was mitigated by design. The Ingress Controller modules of AWS/OCI/GCP do not house permissive rules in Sovereign Pair.

### 3.1 Inbound HTTPS Flow (Internal Loopback)
Tailscale dynamically assigns an A record on the DNS generated by MagicDNS (e.g., `sovereign-oraclenode.tailxxxx.ts.net`).
- **Step 1:** The TLS termination (Handshake) generated via Let's Encrypt ACME happens natively inside the Tailscale *Sidecar* Container on the external network.
- **Step 2:** The decrypted communication enters internally only (Layer 7 local Loopback) from Tailscale's `port 443` to the *Caddy* Web Server's `port 80`.
- **Step 3:** *Caddy* distributes atomic processing internally: The `/api` URI reverts to the Python API and the static assets hit the Nginx Frontend.

### 3.2 Mobile "Headless" Flow and DNS Resolution (NextDNS/DoH)
End-User devices equipped with Private DNS interceptors (such as NextDNS acting under the *Android Private DNS Engine*) will block *ts.net* domains, as it refuses to delegate requests to the Internal DNS provided by the local Tailscale *100.100.100.100* interface.

**SRE/Admin Procedure (Definitive Solution):**
Add a fixed *Custom Rewrite* directive on your DNS Firewall panel (NextDNS Web Admin).
- **Target Record (`A`):** `[machine_magic_dns].ts.net`
- **Target IP:** Machine Address `100.x.x.x` (Local Private IP forged by Tailscale).
This way the NextDNS cloud resolver delivers the Cibrid tunnel directly inside the client without breaking the Mobile's DoT/DoH Security Chain.

## 4. Direct Ignition Instructions

### IaC Parameterizations (Terraform)
1. Modify the `infra/terraform/terraform.tfvars` file.
2. Include your hard credentials (OCIDs, Compute Public RSA, Region, and Availability Domain).
3. Execute the Tofu plan validations:
```bash
tofu init
tofu plan -out=tfplan
tofu apply tfplan
```

### VPN Sidecar Configurations (`.env`)
Every Container needs the Identity Variable to provision the M2M node on the Subnet.
1. Generate a High-Availability *Reusable Ephemeral Key* on the Tailscale Administrative Web Dashboard.
2. Allocate the unified string to the `TS_AUTHKEY` directive in your root Compose `.env`.
3. Raise the bridges:
```bash
docker compose up -d
```
